# 16. 대시보드 해석 가이드 & 유지보수 플레이북

> 2026-04-21 작성. Grafana 대시보드(`hpc-overview`)의 각 패널 수치를 어떻게 읽고, 어떤 대응을 해야 하는지 정리한 문서.

---

## 1. 모델별 스코어 의미

각 모델은 0.0 ~ 1.0 범위의 **이상 스코어(anomaly score)**를 낸다. 절대 확률이 아니라 "해당 모델이 본 시계열이 얼마나 비정상으로 보이는가"를 나타낸다. 앙상블은 이 4개 스코어를 가중합한다.

### 1.1 Chronos (가중치 0.25)

- **역할:** 향후 1시간(64분) CE 시계열을 **예측**. 예측된 피크가 최근 평균보다 얼마나 큰지로 이상도 산출.
- **계산:** `min(1, peak_predicted / (mean_recent + ε) / 20)` — 피크가 평균의 20배면 1.0.
- **값의 의미:**
  - `0.0` : 평탄/감소 예측 — 급증 가능성 낮음
  - `0.3 ~ 0.6` : 완만한 상승 예측
  - `0.7 ~ 1.0` : 큰 급증 예측 (DIMM 열화 진행 중일 수 있음)
- **주의:** CE 시계열이 과거에 전부 0이면 평균이 0 근처라 값이 튈 수 있음 → 1.0으로 포화 가능.

### 1.2 MOIRAI (가중치 0.15)

- **역할:** Zero-shot 시계열 이상탐지. 예측 sample들의 표준편차를 이상 신호로 사용.
- **계산:** `min(1, forecast_std / 10)` — 예측 분산이 클수록 모델이 확신하지 못한다는 뜻.
- **값의 의미:**
  - `0.0 ~ 0.3` : 예측 확신도 높음 (안정 패턴)
  - `0.5 이상` : 예측 편차 큼, 패턴이 흔들림
- **현재 상태:** `uni2ts` 의존성 미해결로 dummy(0.5) 고정 운영 중.

### 1.3 XGBoost (가중치 0.35, 가장 높음)

- **역할:** 통계 피처 기반 지도학습 분류기. **직접적인 장애 확률**을 산출.
- **계산:** `predict_proba(features)[0][1]` — 학습된 모델이 본 최근 구간 피처로 추정한 장애 확률.
- **값의 의미:**
  - `0.0 ~ 0.3` : 정상 패턴과 일치
  - `0.3 ~ 0.7` : 경계 영역
  - `0.7 이상` : 학습 데이터 기준 장애 전 패턴에 근접
- **현재 상태:** 피처 주입 미연결로 0.5 고정. 해결하면 가장 큰 영향(가중치 0.35).

### 1.4 AnomalyTransformer (가중치 0.25)

- **역할:** 최근 100개 CE 샘플의 **재구성 오차(reconstruction error)** 기반 이상탐지.
- **계산:** `min(1, log1p(rec_error) / log1p(threshold))` — 학습된 정상 패턴을 재구성할 수 없으면 이상.
  - **log1p 변환** 적용 이유: 시뮬 데이터에서 서버 간 rec_error가 ~1000배 차이까지 벌어져 단순 비로는 포화됨. log scale로 압축해야 4서버 부드럽게 차별화.
- **값의 의미:**
  - `0.0 ~ 0.3` : 과거 학습된 정상 패턴과 일치
  - `0.5 이상` : 과거에 없던 패턴
  - `1.0` : threshold 초과 — 명백한 이상
- **운영 보정:** threshold는 환경변수 `AT_THRESHOLD`로 오버라이드 가능 (`k8s/rayserve/raycluster.yaml`). 2026-04-30 기준 2500 — vmgnode26 평균 rec_error에 정렬하여 위험 서버를 1.0으로 표시.

## 2. 앙상블 계산식

```
failure_probability = 0.25 × chronos
                    + 0.15 × moirai
                    + 0.35 × xgboost
                    + 0.25 × anomaly_transformer
```

모든 모델이 dummy(0.5)면 앙상블은 0.5 → NORMAL. 실제 CE 데이터가 유입될수록 각 모델이 자기 해석을 내고 값이 흩어진다.

## 3. 종합 퍼센티지별 유지보수 플레이북

| 구간 | 레벨 | 자동 대응 | 운영자 대응 |
|---|---|---|---|
| `0.00 ~ 0.30` | **RECOVERY** | ESXi Maintenance Mode 해제, 복구 알림 | 해당 서버의 vMotion 복귀 가능. 로그 확인 후 특이사항 없으면 정상 라인업 투입. |
| `0.30 ~ 0.65` | **NORMAL** | 없음 | 주기적 관찰. 대시보드 Trend에서 상승세만 확인. |
| `0.65 ~ 0.85` | **WARNING** | ESXi Admission Control → **신규 VM 배치 차단**, Slack 알림 | 아래 "WARNING 대응 절차" 참조. 24시간 내 조치. |
| `0.85 ~ 1.00` | **CRITICAL** | ESXi **Maintenance Mode 자동 전환** + 긴급 알림 | 아래 "CRITICAL 대응 절차" 참조. 즉시 조치. |

### 3.1 WARNING 대응 절차 (24시간 이내)

1. **원인 모델 확인**
   - Chronos가 높음: 향후 CE 급증 예상 — 선제 조치 고려.
   - XGBoost가 높음: 과거 장애와 유사한 피처 패턴 — 가장 신뢰도 높은 신호.
   - AnomalyT가 높음: 최근 CE 패턴이 학습된 정상과 다름 — 새 워크로드일 수도, 열화 시작일 수도.
   - MOIRAI가 높음: 패턴 예측이 불안정 — 부수 신호.
2. **ESXi vmkernel_error_cnt Trend 확인** — 동반 상승이면 실제 장애 전조.
3. **운영 액션**
   - 해당 ESXi의 중요 VM을 다른 호스트로 **수동 vMotion**.
   - BIOS/IPMI 로그에서 DRAM CE 상세 확인 (DIMM 슬롯 식별).
   - 정기 유지보수 창구에 **DIMM 교체 티켓** 등록.
4. **재평가:** 1시간마다 대시보드 확인. 스코어가 0.65 아래로 떨어지면 관찰만, 0.85 돌파하면 즉시 CRITICAL 대응.

### 3.2 CRITICAL 대응 절차 (즉시)

1. **자동 대응 확인** — 해당 ESXi가 Maintenance Mode로 들어갔는지, VM들이 정상 vMotion 되었는지 Slack/vSphere에서 확인.
2. **실패 시 수동 조치**
   - vCenter/ESXi에서 수동 Maintenance Mode 진입.
   - vMotion 실패 VM은 수동 shutdown → 다른 호스트에서 재기동.
3. **하드웨어 점검**
   - ESXi SSH로 `esxcli hardware memory get`, dmesg, vmkernel.log 수집.
   - IPMI SEL 로그에서 UE/CE 이벤트 확인.
   - 유력 DIMM 위치 식별 → **즉시 교체 또는 비활성화(Memory Mirroring)**.
4. **사후 조치**
   - 교체 후 ESXi 재기동 → 72시간 관찰.
   - 스코어가 RECOVERY(0.30 이하)로 떨어지면 서비스 복귀.

## 4. 대시보드 패널 해석 가이드

### 4.1 상단 서버별 "Failure Prob" (게이지/숫자)

현재 순간의 앙상블 값. 위 표의 구간별로 배경색이 바뀌도록 설정되어 있음.

### 4.2 "Models" 패널 (모델별 4개 숫자)

해당 서버에서 마지막 추론의 모델별 스코어. 0.5가 많으면 해당 모델이 아직 제대로 활성화되지 않았다는 뜻(MOIRAI/XGBoost는 현재 dummy 상태).

### 4.3 "vmkernel" (카운터)

ESXi SSH가 읽어온 vmkernel 에러 수. 급격히 증가하면 하드웨어/드라이버 이슈 가능성.

### 4.4 "CPU / MEM" (사용률)

ESXi 호스트 수준 사용률. 장애 예측과 직접 상관은 낮지만 부하 상황 이해에 사용.

### 4.5 하단 Trend 패널들

- **Failure Probability Trend:** 전 서버의 앙상블 스코어 시계열. WARNING/CRITICAL 임계선 가시화 필요.
- **Model Scores Trend:** 모델별 스코어 변화. 어느 모델이 앙상블 상승을 주도하는지 판단용.
- **ESXi vmkernel Errors Trend:** 에러 누적 추이.

## 5. 유의사항

- **현재 CE 데이터는 시뮬레이션.** `k8s/cronjobs/ce-simulator.yaml`가 만들어낸 합성 시계열이며, vmgnode26은 일부러 시간에 따른 degradation ramp를 가진다. 실측 EDAC 수집기 연결 전까지는 **정확한 장애 예측이 아님**.
- **XGBoost/MOIRAI/AnomalyT threshold 이슈**가 해결되기 전까지 앙상블 값은 체계적 편향(주로 0.5 근처)을 가진다. 절대값보다 **트렌드 변화**에 주목할 것.
- CRITICAL 자동 대응은 아직 운영 투입 전이면 **비활성화 상태**일 수 있다. `k8s/cronjobs/esxi-response.yaml` 설정 확인 필요.
